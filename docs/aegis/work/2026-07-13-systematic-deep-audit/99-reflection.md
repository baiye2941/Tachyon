# 审计复盘

## 完成的目标

- 在固定源码基线下，以只读方式完成安全、并发、网络/协议、I/O/持久化、性能、测试/基准、依赖/架构、前端/Tauri、CI 和文档的系统审计。
- 对严重和高风险结论进行了独立交叉复核，并把动态环境不能覆盖的结论降级为条件成立或未验证。
- 产出中文完整报告和可机器读取的外部证据台账。

## 有效做法

1. 先固定 Git 基线、执行范围和非目标，再分流审查，避免把静态判断误报为真实网络复现。
2. 对高风险项同时检查实现、调用路径和现有测试是否真的触发该路径；例如工作窃取测试虽通过，但分片尺寸不足以触发拆分。
3. 将规范、竞品自述、源码和本地命令分开保存；竞品页面只作能力对照，不作性能证据。
4. 用无重试 nextest、基准冒烟、前端构建和覆盖率尝试区分“基础验证已过”与“发布边界未闭合”。
5. 证据 JSONL 显式使用 UTF-8，避免 Windows 默认代码页损坏后续可读性。

## 需要避免的做法

1. 不应把 Loopback、内存存储或缓冲写入基准称为公网、真实磁盘或 BT 性能。
2. 不应因测试名称声称覆盖某功能；必须核对触发条件与实际输入尺寸。
3. 不应以 CI 重试、`continue-on-error`、忽略目录或排除 unsafe 路径代替修复。
4. 不应在未取得 Linux、真实群集、断电或原生 GUI 证据时宣称“已复现”或“已安全”。

## 后续建议

后续实现应从报告第十五节第一阶段开始，先写失败回归，再做最小修复。涉及内存所有权、工作窃取、HTTP 版本完整性、持久化路径或代理边界的改动属于跨模块契约变更，应重新建立计划、单独进行代码审查，并在目标平台执行专项验证。
