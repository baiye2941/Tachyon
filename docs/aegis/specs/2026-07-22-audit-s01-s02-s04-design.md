## 3. S-01：默认拒绝受限 BT peer，显式 opt-in 放行

### 3.1 不变量与 canonical owner

```text
magnet URI pe= ───────────────────┐
                                  │
MagnetConfig.peer_addrs ──────────┼──> protocol canonical collection
                                  │        ├─ core 无 feature 例外的地址分类
                                  │        ├─ stable dedup / source quota / total cap
                                  │        └─ 仅在此处生成 AddTorrentOptions.initial_peers
                                  v
                            librqbit session add
```

- `tachyon-core`：保存 `allow_private_peers`、提供无 test-harness 放宽的受限 IP 纯分类、保持配置可加载。
- `tachyon-protocol`：所有 `MagnetProtocol` cache-miss add 路径在紧邻 `AddTorrentOptions.initial_peers` 的唯一位置收集/过滤 peer。不得让 `add_magnet_to_session` 接受调用方任意构造的裸 `Vec<SocketAddr>`；它必须接收 URI 与 `MagnetConfig`，或接收只能在本模块构造的私有 normalized input。
- `tachyon-app`：复用已有 `magnet_changed` → `BtSession` rebuild 事务；配置成功后影响后续 BT session/连接，不承诺改写已运行 torrent。
- frontend：只负责 DTO、草稿、patch、风险可见性；不自行执行安全策略。

### 3.2 地址分类与 test-harness 边界

现有 `reject_forbidden_ip` 在 `test-harness` 下为了 HTTP wiremock 放行 loopback，不能直接作为 BT policy 的最终 oracle。

在 `tachyon-core::safety` 增加一个无 feature 例外、私有规则单一来源的基础分类 API（纯 predicate 或分类返回值）。它与现有 HTTP API 的关系：

- HTTP `reject_forbidden_ip` 保持 test-harness 行为；
- BT policy 和配置诊断调用新分类 API，因此生产与 test feature 下都一致拒绝 loopback、private、link-local、unspecified、multicast、reserved、CGNAT、documentation、benchmark 等受限地址；
- 不复制 CIDR/IP 规则表，不在 protocol 维护另一套地址范围。

该 helper 是为消除已有 core 安全分类在 feature 下的语义分歧而增加的复用面，不是新安全 owner。

#### S01-B：protocol collector slice card

本卡片是既有 S-01“默认拒绝、显式 opt-in”契约的实施切片细化；不新增或改变已批准的产品行为。

- **私有 collector 合同**：在该模块定义且保持 module-private：


### 3.3 输入语义

- URI `pe=`：保持容错。不可解析、端口 0、默认策略受限、超限项都不进入 collection；日志只记来源与计数摘要，不包含 URI 或 SocketAddr。
- 配置 `peer_addrs`：明确固定为 IP literal `SocketAddr`（`IP:port` / `[IPv6]:port`），本切片不新增 hostname/DNS 支持。空白、端口 0、不可解析、默认策略受限项保留在旧配置中但默认不使用；不能让 `AppConfig::validate()` 因它们失败、不能自动删除。
- `allow_private_peers=false`：所有受限地址从 URI/config collection 排除。
- `allow_private_peers=true`：可解析且端口非零的受限地址可进入 collection；非法/端口 0仍排除。
- 配置字段、前端 hint 与 i18n 必须修正为 IP literal 语义，不能继续声称泛化 `host:port`。

### 3.4 有界、稳定、公平的 collection

当前 librqbit 8.1.1 将 `initial_peers` 直接转为 stream，未提供该输入的上限保证。因此在 Tachyon 边界定义独立 anti-abuse 限制：

- `MAX_INITIAL_PEERS = 64`：总量；不与 `MAX_CACHED_HANDLES` 共用或类比。
- 每个来源先各自稳定去重，再按 URI 最多 32、配置最多 32 的可预测配额加入。
- 若一方未用满，其剩余容量允许另一方补足到总 64。
- 跨来源重复地址保留 URI 的首次出现位置；每来源内部保持 first-seen 顺序。
- 记录 URI/config 的非法、受限、重复、超限计数摘要；不得记录地址值、完整链接或其他配置。

## 4. S-02：未来 schema 快照拒绝、隔离与全流程 fail-closed

### 4.1 分类模型

future schema 是有效但当前版本不支持、必须保护的数据，不能进入 `corrupt_keys`。

扩展 `RecoveryResult`：

- `corrupt_keys`：无效 JSON、非法 header、无法解析当前/legacy 内容；
- `unsupported_schema`：包含 key、发现版本、当前支持版本的受保护记录；不包含 raw JSON、URL 或 save path。

`TaskStore`、启动恢复、日志和 task command 调用方必须保留类别；future schema 的日志只能表述“已隔离，需要较新版本”，不得称“损坏”。此为现有 RecoveryManager 的结果模型扩展，不新增另一套恢复 manager。

### 4.2 header guard 的唯一算法

`RecoveryManager::load_task_snapshot_by_key` 是唯一解析模型。header guard 必须使用一次独立的顶层 JSON object streaming `Deserializer` / `Visitor`：逐字段读取、计数 `schemaVersion`、以 `IgnoredAny` 消费其余值。不得先反序列化到 `serde_json::Value`/Map，因为该路径会折叠重复 key，无法兑现重复 header 的 fail-closed 合同。

精确顺序：

```text
get_raw(key)
  -> streaming parse top-level object header
  -> non-object / duplicate schemaVersion / null / string / negative / overflow: InvalidData，不 fallback
  -> schemaVersion missing: legacy candidate，允许 TaskSnapshot 再 TaskRecord fallback
  -> schemaVersion > CURRENT: Unsupported，不 TaskSnapshot / TaskRecord fallback
  -> schemaVersion <= CURRENT: TaskSnapshot parse；必要时既有 legacy fallback
```

header guard 必须早于任何 `TaskSnapshot` / `TaskRecord` serde fallback；不能让 malformed header 靠 serde default 降格为 legacy。错误消息只包含版本/key 的最小诊断，不包含 raw JSON、URL、路径。

### 4.3 Store 内的线性化保护与 restore 原子性

`RecoveryManager` 已有 `progress_lock`，所有 task snapshot mutation 必须在同一锁内完成“guard → mutation”。对于磁盘已有 future record：

- `load_task_snapshot`、`save_task_snapshot`、`update_snapshot`、`restore_task_snapshot`、`remove_task` 都返回 `ErrorKind::Unsupported`，raw bytes 不变。
- `restore_task_snapshot` 不得采用“guard 成功 → 删除 tombstone → 再普通 save”的顺序。它必须在同一 `progress_lock` 下：先 header/revision guard；再走一个仅供 restore 使用、临时绕过 tombstone 拒绝但仍执行 revision/CAS 的 durable write；**仅当 durable write 成功后**删除 tombstone。guard、serialization、I/O、fsync 或 rename 任一失败均保留 tombstone；旧 revision save 仍不能复活任务。
- `load_all_task_snapshots`、`recover_pending_snapshots` 和旧 `recover_pending_tasks` 必须有一致的 future-schema 行为：前两者继续处理合法项并累积 `unsupported_schema`；旧 API 要么重定向到新结果模型的兼容 adapter，要么明确仅向其直接调用方返回 `Unsupported`，但不得在未分类状态下将整批合法恢复阻断。当前 workspace 调用方不得依赖该旧 API。

### 4.4 App 破坏性工作流的 canonical preflight

“扫描是否有 future record”与 mutation 分离会形成 TOCTOU，不能满足用户选择的全部 fail-closed。`TaskStore` 必须提供一个由 RecoveryManager 同一 `progress_lock` 支撑的、结构化 protected-record preflight / operation guard：对 task-key namespace 的扫描与下一步 snapshot mutation 处于同一线性化边界；不能由 app 先 `keys()` 再无锁删除。

调用顺序固定为：

1. **单任务删除**：`TaskService::delete_task` 在 local file 删除、repository removal、undo record mutation 与 `remove_snapshot` 前 await protected-record guard。不得使用 `.ok().flatten()` 吞掉 load error，也不得把 `Unsupported` 写 warning 后继续。
2. **导出**：在创建临时文件前通过 TaskStore 的结果模型检查；存在 `unsupported_schema` 时失败，不生成不完整 backup。
3. **覆盖导入**：解析并验证 backup（包括每个 task snapshot schema）后，先在同一 guard 下检查本地 task namespace；只有所有 local/backup item 都可操作，才持久化 config、改 repository、删除旧 snapshot。
4. **非覆盖导入**：先验证所有 backup task schema，并同步预检每个目标 key；全部通过后才插入 repository，并 await 每个 snapshot save。禁止 detached `spawn_blocking` 吞掉 save failure。
5. **导入 future backup**：任一 task `schema_version > CURRENT` 时，在任何 mutation 前失败。

`unsupported_schema` 必须从 `RecoveryResult` 经 `TaskStore`、startup recovery、task service 与 task commands 传播为“需要较新版本”的可操作错误；future schema 不能落入 corrupt JSON 文案。

如果实现需要新的持久化格式、第二个 schema owner、跨层反序列化 adapter，或无法在 `progress_lock` 下提供线性化 guard，停止并回到设计审查。

### 4.5 S-02 验收

1. future raw JSON 单项 load 是 `Unsupported`，错误不泄漏 URL/path/raw JSON。
2. streaming header parser 拒绝 non-object、重复 key、null/string/negative/overflow；legacy 缺字段和合法 current schema 的 fallback 保持。
3. save/update/restore/remove 都 fail-closed，raw bytes 恒等；测试覆盖 guard 成功但 durable write 失败后 tombstone 仍生效。
4. 所有 batch recovery API 的 future 行为已定义；新结果 API 下合法记录继续可用，future key 进入 `unsupported_schema`，不进入 `corrupt_keys`。
5. export、single delete、overwrite import、non-overwrite conflict import、future backup input 都在 config/repository/snapshot/local-file mutation 前 fail-closed，并断言出错后 config bytes、repository、raw snapshot、tombstone 不变。
6. TaskRecord legacy fallback、revision/tombstone、正常 backup 路径回归绿色。

### 4.6 修复与退役

- **Repair track**：streaming header guard 阻断宽松 future parse；受保护结果类别与线性化 destructive guard 阻断 future 数据丢失。
- **Retirement track**：退役 future-as-corrupt、备份静默遗漏、restore 先清 tombstone、app 先 mutate 再 warning 的路径。
- **Residual risk**：旧客户端无法显示/恢复 future task；这是用户选择的安全降级。raw passthrough backup 是明确未实现的后续产品设计。

## 5. S-04：PlanComplete blocking read、顺序屏障与最小测试接缝

### 5.1 行为合同

- 收到 `PlanComplete` 后，worker 必须在消费同一 `progress_rx` 的下一事件前 await 完整 snapshot load 并应用 seed/fallback；禁止 fire-and-forget。
- 读取本体必须在 `tokio::task::spawn_blocking` 内；成功、None、store error、JoinError 四种结果显式处理。
- JoinError 不 panic，fallback 到 repository downloaded。
- callback 顺序不变：PlanComplete 的 `delta=None` 必在后续 Chunk/Started callback 前完成。
- 为保持最小改动，读取前已经存在的 `FragmentStateStore::init` 与本地 `completed` 集合初始化可以保留；“屏障前不可观察 state”仅指依赖 snapshot seed 的 `TaskInfo.downloaded` / `progress` / `fragments_done` 与 PlanComplete callback。测试不得把既有 fragment topology 初始化误判为顺序回归。
- `TaskStore` 仍为同步 API；不推广 async store，不改 snapshot 格式，不修改已经正确的 `task_service`。

### 5.2 最小 test-only 接缝

固定采用一个只在本模块使用的 `load_plan_snapshot(task_store, task_id)` production helper；它在 production 中唯一调用 `spawn_blocking(move || task_store.load_snapshot(&task_id))`。`cfg(test)` 的 hook **位于该 blocking closure 内**，因此 hook panic 会变成可验证的 `JoinError`，而不是展开 chunk-reader async worker。

hook 必须具备以下确定形态：

1. 模块私有 `test_hook`，不出现在跨 crate 公共 API；类型为一次性 blocking closure，输入 `Arc<TaskStore>` / `String`，输出与真实 `load_snapshot` 相同的 `Result<Option<TaskSnapshot>, AppError>`。
2. RAII guard 安装和 drop 自动复位；一个静态测试锁串行化安装，禁止并行测试串扰。
3. hook 通过 `started_tx` 发送“blocking closure 已开始”，然后等待 `release_rx`；release 后返回指定 snapshot/store error，或在 closure 内 panic 以产生 JoinError。
4. reactor 测试明确使用 `#[tokio::test(flavor = "multi_thread", worker_threads = 1)]`。收到 `started_rx` 后，在未释放 loader 栅栏时 spawn sibling Tokio task 并等待其 oneshot；该 sibling 必须完成，才能证明唯一 reactor worker 未被同步 read 占住。
5. 测试使用 oneshot/channel 状态和 `done_rx.await`；不得用 sleep、polling 或 timeout 作为业务断言。

该 hook skeleton 属于已批准的 test infrastructure，Tester 拥有它和全部断言资产；Coder 只拥有 production helper 和 PlanComplete wiring，不得先写测试断言或以接口占位替代业务 RED。

### 5.3 S-04 验收

1. 栅栏未释放时 sibling Tokio task 能完成；依赖 seed 的 TaskInfo state 与 PlanComplete callback 不得提前，且同一 progress channel 后续 Chunk 不得被消费。
2. release 后，PlanComplete seed、`delta=None` callback、随后 Chunk 的 seed+delta 和 callback 顺序确定。
3. store business error 与 closure panic 产生的 JoinError 都 fallback 到 repository downloaded；不 panic、不丢 completion signal。
4. 既有 snapshot seed、partial、失配、无 snapshot tests 保持绿色；新增测试只用 channel/oneshot completion，不复制 sleep polling。

### 5.4 修复与退役

- **Repair track**：用 await 的 blocking dispatch 取代 async worker 内同步读。
- **Retirement track**：删除 PlanComplete 直接调用同步 `load_snapshot` 的唯一位置；不保留并行读取 fallback。
- **Residual risk**：最小 hook 只证明此 branch 的 dispatch/ordering，不能作为所有 store 调用均已 async-safe 的声明。
